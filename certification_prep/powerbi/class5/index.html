Workspace Management

Workspaces are collaborative environments in the Power BI service where you and your colleagues can work together on datasets, reports, dashboards, and dataflows.

Creating & Managing Workspaces

Purpose: Workspaces facilitate content organization, collaboration, and staged deployment (e.g., Dev, Test, Prod workspaces).

Workspace Types (Focus on New Experience/v2):

  • My Workspace: A personal workspace for individual users. Content here cannot be shared in the same way as in collaborative workspaces and cannot be used to create Apps.
  • Collaborative Workspaces (New Experience/v2): The standard for teamwork.
    • Created in the Power BI service.
    • Offer granular roles (Admin, Member, Contributor, Viewer).
    • Can be assigned to a Premium capacity for enhanced features and performance.
    • Allows creation of Power BI Apps for content distribution.
  • Classic Workspaces (v1): Older type, often tied to Microsoft 365 groups. While you might encounter them, the exam focuses on the new experience.

Creating a New Experience Workspace:

  1. In Power BI service, navigate to "Workspaces" -> "Create a workspace".
  2. Provide a unique name, description (optional).
  3. Advanced settings: Assign to Premium capacity, set contact list, choose default storage format (for large datasets).

Roles & Permissions within Workspaces

Access to content within a new experience workspace is managed through four primary roles:

  • Admin: Full control. Can add/remove users (including other admins), publish, unpublish, update app, delete workspace, manage all content.
  • Member: Can publish content, share items (if allowed by admin settings), manage gateway connections, schedule refresh, create/update app (if allowed). Can add users with Member, Contributor, or Viewer roles (but not Admins).
  • Contributor: Can create, edit, and delete content within the workspace (datasets, reports, dashboards). Can publish reports. Cannot publish or update an app, or share content broadly (unless they own it and sharing is permitted).
  • Viewer: Can view and interact with reports and dashboards. Cannot modify content. Can read data stored in workspace dataflows. Requires a Pro license or content to be in Premium capacity.

Users or security groups (Microsoft 365 groups, distribution lists, mail-enabled security groups) can be assigned these roles.

It's best practice to assign roles based on the principle of least privilege.

Exam Tip: Understand the differences and capabilities of the four main workspace roles (Admin, Member, Contributor, Viewer). Know that "My Workspace" has limitations for sharing and app creation. Focus on the "new workspace experience."

Dataset Management in the Service

Once published, datasets in the Power BI service require ongoing management to ensure data accuracy, security, and discoverability.

Publishing Datasets & Reports

The primary way to get your data model and reports into the Power BI service is by publishing from Power BI Desktop.

  • Process: In Power BI Desktop, click "Publish" on the Home ribbon. Select a destination workspace.
  • What's Published: The entire .pbix file's content is uploaded. In the service, this typically results in:
    • A Dataset: Contains the data model (tables, relationships, measures, calculated columns), connection information, and refresh settings.
    • A Report: Contains the visuals and pages you designed.
  • The dataset and report become separate, manageable items in the service. Multiple reports can be built from a single published dataset.

Scheduled Refresh & Gateways

Keeps your dataset data current in the Power BI service.

Configuring Scheduled Refresh:

  • Done in the dataset settings in the Power BI service.
  • Set refresh frequency (e.g., daily, weekly) and specific times (up to 8 timeslots per day for Pro, 48 for Premium).
  • Specify time zone.
  • Configure refresh failure notifications to dataset owners.

On-premises Data Gateway:

  • Purpose: Acts as a secure bridge between the Power BI service (cloud) and your on-premises data sources (e.g., SQL Server on a local server, local files).
  • Types:
    • Standard mode (recommended): For enterprise use, can be shared by multiple users, supports DirectQuery and scheduled refresh for various services. Installed on a persistent server.
    • Personal mode: For individual use, runs as an application, only supports import, cannot be shared. (Less focus for exam).
  • Configuration: Install gateway software, register it, configure data sources within the gateway in the Power BI service, map datasets to use the gateway connection.

Cloud-based data sources (e.g., Azure SQL Database, SharePoint Online) generally do not require a gateway and can be refreshed directly by the service if credentials are configured.

Dataset Security: Row-Level Security (RLS)

RLS restricts data access for given users at the row level. Filters are applied automatically based on the user's identity.

Static RLS:

  • Define roles and DAX filter rules directly in Power BI Desktop (Modeling tab -> Manage roles).
  • Example rule for a "RegionalSales" role: [Region] = "North".
  • After publishing, in the Power BI service (Dataset settings -> Security), assign users or security groups to these roles.

Dynamic RLS:

  • DAX filter rules use functions like USERNAME() or USERPRINCIPALNAME() to filter data based on the logged-in user.
  • Typically involves a separate mapping/security table that links user principal names to the data attributes they are allowed to see (e.g., a table linking UserEmail to SalesRegion).
  • Example rule: [UserRegion] = LOOKUPVALUE(UserSecurity[Region], UserSecurity[UserEmail], USERPRINCIPALNAME()).

Testing RLS: Use the "View as roles" feature in Power BI Desktop (Modeling tab) or in the Power BI service (Dataset security) to test how data appears for different roles/users.

Endorsing Datasets (Promoted, Certified)

Endorsement helps users find high-quality, trustworthy datasets and encourages their reuse across the organization.

  • Promoted: Dataset owners or workspace members with write permissions can promote datasets they deem valuable and ready for wider use. It's a self-service endorsement.
  • Certified: A higher level of endorsement indicating the dataset meets an organization's quality and governance standards. Certification is a more controlled process; typically, only authorized administrators or a central governance team can certify datasets. (Power BI admin must enable dataset certification and define a process).

Endorsed datasets are highlighted in the Power BI service (e.g., in the "Get Data" experience when connecting to Power BI datasets), making them easier to discover.

This helps build a "single source of truth" and reduces data duplication.

Exam Tip: Know the steps to publish and configure scheduled refresh. Understand the critical role of the on-premises data gateway (standard mode). Differentiate static vs. dynamic RLS and how they are implemented. Recognize the purpose of "Promoted" and "Certified" endorsements.

Sharing and Distributing Content

Power BI provides multiple ways to share your insights, from direct sharing of reports and dashboards to packaging content into formal Apps.

Sharing Reports & Dashboards

Allows ad-hoc sharing of individual items with specific users or groups.

  • How: Use the "Share" button on a report or dashboard in the Power BI service.
  • Permissions for Recipients:
    • Allow recipients to re-share the item (if they have Pro license).
    • Allow recipients to build content with the underlying dataset(s) (Build permission).
    • Send an email notification.
  • Links: Can also share via a link. Be mindful of link settings (e.g., "People in your organization," "Specific people").
  • Licensing: Generally, both the sharer and the recipient need a Power BI Pro license. If the content is hosted in a Premium capacity, recipients with Free licenses can view it (if granted access).

Manage access via the "Manage permissions" pane for the item.

Creating & Managing Apps

Power BI Apps provide a more formal and structured way to distribute a collection of related content (dashboards, reports, datasets, workbooks) to a broad audience.

  • Creation: Created from a workspace by users with Admin, Member, or Contributor (if allowed) roles. Click "Create app" or "Update app" in the workspace.
  • Configuration:
    • Setup: App name, description, logo, theme color.
    • Navigation: Define the app's navigation structure (sections, links to specific content items). Choose which reports/dashboards to include.
    • Permissions (Audiences): In new experience workspaces, you can define multiple audiences for an app. Each audience can see a different subset of the app's content and have different permissions. This allows tailored content delivery from a single app.
  • Publishing/Updating: After configuration, publish the app. Updates to workspace content are not automatically reflected in the app; you must "Update app."
  • Consumption: Users can find and install apps from "Apps" in the navigation pane or via a direct link.

Apps are generally the preferred method for broad content distribution within an organization.

Exporting Data from Visuals & Reports

Power BI allows users to export data, subject to permissions and settings.

  • From Visuals:
    • Click the "..." (More options) on a visual.
    • Export data:
      • Summarized data: Exports the aggregated data shown in the visual.
      • Underlying data: Exports the detailed data rows that make up the visual (if allowed by report settings and dataset permissions).
    • Formats: Typically .xlsx (Excel) or .csv.
  • Analyze in Excel: Users can connect to a Power BI dataset from Excel (using the Power BI publisher for Excel or Get Data from Power BI dataset) to create PivotTables or other analyses directly in Excel, leveraging the live Power BI dataset. Requires Build permission on the dataset.
  • Export Report:
    • To PDF (static snapshot).
    • To PowerPoint (static images of pages, or live pages if embedding is enabled).

Control: Power BI admins can control data export settings at the tenant level. Report creators can also influence some export options (e.g., for specific visuals).

Exam Tip: Differentiate between sharing individual items and distributing content via Apps. Understand App audiences. Know the options for exporting data (summarized vs. underlying) and the "Analyze in Excel" feature. Licensing implications for sharing (Pro vs. Premium) are also important.
